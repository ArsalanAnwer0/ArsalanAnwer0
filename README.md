# I learn things by breaking them into:

## **what · how · why**

you can learn about me the same way

## what

I build things that run and make sure I understand why they run. Mostly backend systems, infrastructure, and whatever sits between code and reality. Lately that’s meant building backend services (mostly in Go), containers, Kubernetes deployments, and cloud infrastructure. 

## how
- design services that are simple, predictable, and easy to reason about
- make it reproducible from scratch  
- assume it will break, and design for that  

## why
because systems that only work once aren’t useful.

If I don’t understand how something works, I don’t trust it.  
If I have to do something twice, I’d rather automate it.
