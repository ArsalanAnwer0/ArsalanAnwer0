# I understand things by breaking them into:

## **who · what · how · why**

so here’s the same, applied to me

## who
I build things that run and make sure I understand why they run.

Mostly backend systems, infrastructure, and whatever sits between code and reality.  
If it deploys, scales, or breaks, I’m interested in it.

## what
Lately that’s meant building backend services (mostly in Go), containers, Kubernetes deployments, and cloud infrastructure.

A lot of my time goes into setting things up, tearing them down, and rebuilding them properly  
until they’re not just working, but understandable and reproducible.

## how
- design services that are simple, predictable, and easy to reason about
- make it reproducible from scratch  
- assume it will break, and design for that  

## why
because systems that only work once aren’t useful.

If I don’t understand how something works, I don’t trust it.  
If I have to do something twice, I’d rather automate it.
